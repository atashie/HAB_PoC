# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

This repository is a **proof-of-concept Harmful Algal Bloom (HAB) diagnostic & treatment-recommendation tool**, built for the **SePRO Lead Data Scientist finalist case study**. SePRO is a water-resource and plant-protection science company, and this would be their *first* data-science hire — so the work is judged as much on *how* we reason and build as on the result.

The deliverable has three parts, presented in ~45 min to a mixed panel (scientific, platform, commercial, operations) plus ~30 min of Q&A, with a later recorded code walkthrough:

- **Part A — Analysis.** Take a water-health question through the full DS arc (define → acquire → prepare → explore → model → evaluate → communicate → deploy), **fusing a remote-sensing/satellite signal with in-situ aquatic (or weather) data**, explainable and defensible enough to sit behind a **product claim**.
- **Part B — Tool.** A small, working, interactive tool for a **non-technical user** (a SePRO field/technical specialist or customer) that draws directly on Part A — select a water body / conditions → get a risk readout, an action window, or the key drivers.
- **Part C — Prototype → platform.** A short, practical sketch of data architecture, production/monitoring (incl. model drift & auditability), and a 6–12 month plan.

**Authoritative source of truth:** `SePRO/SePRO_Lead_Data_Scientist_Presentation_Brief.pdf` (verbatim text copy at `docs/brief_extracted.txt`). When a requirement is in doubt, re-read the brief — do not rely on paraphrase, including this file.

## Non-negotiable operating principles — READ FIRST

These are the spine of the project. The brief grades on exactly these axes: results that *"hold up to scrutiny,"* an approach *"explainable and auditable"* enough to stand behind a claim, showing *"why the model says what it says, not just what it predicts,"* and *"the rigor and honesty of your modeling and validation."* They are not aspirational — they gate every artifact we produce.

### Fidelity — real data, real results
- **Real, public, cited data only.** Never fabricate data points, numbers, dates, sources, or citations. If a value is unknown, say so — do not invent a plausible one.
- **Synthetic data is allowed only to backstop, and only when labeled.** The brief permits synthetic data when real public data cannot serve. If used it must be (a) clearly justified, (b) generated by checked-in code, and (c) labeled **SYNTHETIC** *everywhere it appears* — code, plots, tables, slides, and the tool. Synthetic data must never masquerade as observed data.
- **Report results as they are — including bad ones.** If a model underperforms its baseline, fails to generalize, or isn't statistically robust, say so plainly. The brief explicitly calls a clear-eyed negative result *"a great outcome."* Weak/negative findings are deliverables, not things to hide or dress up.
- **Every number carries a baseline and an uncertainty.** A metric with no baseline and no error bar / confidence interval is not a finding. Always compare against a naive / persistence / climatology baseline.
- **Actively hunt leakage and overfitting.** Guard against target leakage and temporal (look-ahead) and spatial leakage; validate with splits appropriate to space and time, not random shuffles on autocorrelated data. Document the checks you ran.
- **Correlation ≠ causation.** State it explicitly wherever a driver or treatment implication is drawn.
- **No result-shopping.** No cherry-picking lakes, time windows, or metrics to flatter the story. Pre-specify the evaluation where feasible and report the whole picture.

### Transparency — traceable, no bullshit
- **Every claim is traceable.** Any statement in a notebook, slide, or the tool must trace to (1) the exact data (source + query + access date) and (2) the exact code that produced it, in enough detail to reproduce. **If it can't be traced, it can't be claimed.**
- **Everything is reproducible.** Data access is scripted; pipelines are deterministic (fixed seeds); results regenerate from checked-in code. Prefer scripts/notebooks in the repo over one-off manual steps.
- **Log assumptions and trade-offs as you go.** The brief asks for both. Keep a running decisions/assumptions log; surface shortcuts and limitations rather than burying them — *"noting a known limitation reflects well on you."*
- **Explainability over sophistication.** A well-validated simple model beats an opaque complex one (the brief says so outright). Prefer methods whose reasoning can be shown (feature importance, partial dependence, transparent structure). Always be able to answer *why* a prediction is what it is.
- **Calibrate language to evidence.** No overclaiming in stakeholder-facing text or in the tool's UI. The tool must never imply more certainty than the model supports.
- **Flag AI-shaped judgment.** Where an LLM or coding tool shaped a substantive choice or produced content, note it so a human can verify — the brief asks where our own judgment shaped the result.

### The claim gate
Before any number or statement enters the slides or the tool, it must pass all four:
1. **Traces** to a cited real source (or clearly-labeled synthetic).
2. **Regenerates** from checked-in code.
3. **Reported** with a baseline and an uncertainty.
4. **Bounded** — stated with its limits and where it does *not* apply.

## Scope & ways of working
- **Timebox: ~6–8 focused hours total.** A crisp design + a thin working slice beats an exhaustive build. **Lean toward the analysis and the thinking;** Part B and Part C are lightweight slices that support the story. When the thinking is shown, stop.
- **Verify before claiming.** Run the code and read the actual output before asserting a result is correct or a number is real. Evidence before assertions.
- **Use the process skills.** Brainstorm before design/build; use systematic debugging for any bug; use TDD for tool logic worth trusting. Skills define *how* we work, not just what.
- **Public data only; cite every source** (name, URL, access date, query/params). A running list of sources is itself part of the final deliverable.
- **State assumptions when SePRO-specific detail is missing** — make a sensible one and record it.

## Sanctioned public data sources (from the brief; all public, cite on use)
- **EPA CyAN** — satellite-derived cyanobacteria detection for U.S. lakes (the spectral signal); weekly cyanoHAB forecasting; Sentinel-3 / Landsat links. https://www.epa.gov/water-research/cyanobacteria-assessment-network-cyan
- **Water Quality Portal** — in-situ chemistry from USGS, EPA & 400+ agencies: nutrients, chlorophyll-a, temperature. https://www.waterqualitydata.us/
- **USGS NWIS** — streamflow / gage / water data; `dataRetrieval` packages for R & Python. https://waterdata.usgs.gov/
- **EPA National Aquatic Resource Surveys / National Lakes Assessment** — standardized lake condition & nutrient data. https://www.epa.gov/national-aquatic-resource-surveys
- **NOAA / NCEI** — temperature, precipitation, climate data to pair with water data. https://www.ncei.noaa.gov/
- **State environmental / DEQ lake-monitoring programs** — open bloom advisories & monitoring data.

These are a starting palette, not a mandate — but any source we add follows the same citation + traceability rules.

## Repository conventions (greenfield — to be built)
No application code exists yet. As we build, keep the repo reproducible and legible:
- Scripted, re-runnable data acquisition; cache raw pulls and **never hand-edit raw data**.
- Analysis in notebooks/scripts that regenerate all figures and metrics from source.
- A single source-of-truth list of data citations.
- Clear separation of **raw** (as-downloaded) vs **derived** data.

**Build / lint / test / run commands and the concrete stack are intentionally unspecified until we choose them** (see Open decisions). Add them here the moment they exist — do not leave future instances guessing or inventing them.

## Open decisions (resolve during research / brainstorming — do not pre-commit)
- **Framing** — risk forecasting vs. early-warning anomaly detection vs. driver/treatment analysis. *(Brief's steer: timing & targeting matter most — a right-water-body call two weeks early beats a perfect one made after the bloom is already visible.)*
- **Geography & scope** — which lakes / region and what time span, driven by data availability and coverage.
- **Stack** — analysis language + tool framework (Streamlit / Gradio / notebook-as-app / web). Choose what moves fastest and demos cleanly.
- **Model family** — favor the simplest method that fits the decision and stays explainable.
